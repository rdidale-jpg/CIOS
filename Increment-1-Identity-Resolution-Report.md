# Increment 1 Identity Resolution Report

**Status:** Proposed implementation report for Increment 1.  
**Scope:** UK Banking governed corpus only. Source IDs were not rewritten.

## Existing Identifier Families

- `BK-IND-*`, `BK-REF-*`, `BK-INF-*`, `BK-ENT-*`, `BK-CMP-*`, `BK-FLR-*`: Banking governed asset identifiers from the Banking manifest.
- `EK-BANK-RHYP-*`: Banking Reinvention Hypothesis source asset identifiers.
- `BRH-*`: hypothesis or market-participant intelligence identifiers.
- `BK-OBS-*`: Observation identifiers referenced by existing Banking runtime documentation.
- `SDOC-*`: source document identifiers generated during research migration.
- `BRIEF-*`, `METH-*`: research-package object identifiers appearing across multiple source documents.
- Runtime-only `proj-*`, `ws-*`, `ing-*`, `aud-*`, and `corr-*`: non-canonical fixture/runtime identifiers.

## Collisions

`BRIEF-001` and `METH-001` appear more than once in `enterprise-knowledge/banking/canonical/banking-research-canonical-objects.json` across source documents and object-type contexts. Increment 1 must treat these as collision-prone unless scoped by owning asset, source document, release, and object type.

## Missing IDs

Some relationships in navigation and report prose target concepts such as persona sets, journeys, or methods without separate governed object IDs. These remain unresolved rather than inferred from display labels.

## Unstable IDs

Runtime projection IDs, workspace IDs, ingestion run IDs, audit IDs, and correlation IDs are runtime state, not canonical identity. They must not replace source asset IDs.

## Cross-File Identity Issues

Manifest assets use stable asset IDs, while canonical research migration objects may reuse compact IDs across archive members. The runtime must preserve `source_asset_id`, `source_document_id`, archive member path, release, and version/hash when presenting identity.

## Relationship-Resolution Issues

- Relationship targets such as `EGM-001 — Enterprise Growth Method` include label text and require parsing only as an original identifier, not as a canonical target rewrite.
- Relationship type strings in manifests can be used as governed vocabulary only within source asset scope; otherwise use `unclassified`.
- Cross-enterprise relationships in Increment 1 require both enterprise contexts or a safe-unavailable/incomplete result.

## Recommended Runtime Resolution Envelope

Use `original_identifier`, `original_identifier_type`, `owning_asset`, `resolved_runtime_identifier`, `resolution_status`, `collision_status`, and `resolution_rationale`. This envelope scopes resolution without mutating source IDs.

## Increment 1 Supported Object Types

Supported for read projection: Industry Foundation, Industry Twin, Reference Model, Infrastructure Twin, Enterprise Twin, Differential Matrix, Discovery Register, Reinvention Hypothesis Register, Flora Navigation Specification, Observation references, Evidence/source-document references, Unknown, Contradiction, governed relationship, lineage node, audit event, ingestion report, workspace state, and safe-unavailable response.

## Deliberately Unsupported Object Types

Candidate commercial opportunities, generated claims, Recommendations, scores, actions, hypotheses generated by runtime workers, provider/model artefacts, canonical promotion candidates, human approval tasks, graph-storage internals, and write-back records are outside Increment 1.

## Deferred Migration Issues

- Canonical unification for duplicate compact research IDs.
- Owner-authority completion where manifest owner is `TBD`.
- Formal governed vocabulary registry for relationship types.
- Full Evidence/Observation object materialisation for all Markdown assets.
- Cross-enterprise identity boundary model beyond UK Banking fixture scope.
